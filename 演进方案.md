## 审视方法论

以 `tanweai/pua`（原版 PUA skill，以下简称 **pua**）为竞品基准，逐维度对比 PUAX（MCP 派生分支），找出 pua **相对 PUAX 领先**的能力，据此制定 PUAX 的赶超路线。

**审视原则**：不照搬，取其魂——pua 是"压力修辞 + 方法论"的单 skill 多平台分发体系；PUAX 是"MCP 工具 + 50 角色"的运行时引擎。两者架构不同，演进不是复制，而是把 pua 验证有效的**行为机制**迁移到 PUAX 的 MCP 引擎里。

## 一、两项目定位对比

| 维度 | pua (tanweai) | PUAX |
|------|---------------|------|
| 架构形态 | Skill 插件（SKILL.md + shell hooks + slash commands） | MCP Server（TypeScript 运行时 + 50 skill 文件） |
| 分发方式 | Claude 插件市场 + Vercel Skills CLI + 11 平台原生适配 | npm (`npx puax-mcp-server`) + 平台导出 |
| 核心资产 | 15 企业味道 × 15 方法论 + 22 命令 + 7 hooks + 7 子代理 | 50 角色 × CC-BOS 文言文 + 16 触发 + 推荐算法 |
| 版本 | 3.5.0 | **3.10.0** |
| 语言 | 中/英(PIP)/日 三语 | 中文 + PIP Edition（`language=en`）；日文版暂不实施 |
| 测试 | 17 行为评测脚本 + 18 组对照实验基准 | Jest 578+ + `evals/run-all.js` 12 项守门 + 可选 L4 DeepSeek |

**关键差异（2026-07-04 更新）**：pua 在 slash 命令 UX、评测规模、日文生态上仍有优势；PUAX 在 **MCP 运行时、50 角色矩阵、CC-BOS、混合触发检测、可观测性** 上领先。两者差距已从「PUAX 缺行为闭环」转为「体验细节与生态规模」层面的迭代。

### 🔴 P0 级差距（核心能力缺失，直接影响有效性）

#### 1. 方法论智能路由（Methodology Router v3）
**pua**：任务类型 → 自动选最优味道 + 方法论；连续失败 → 按失败模式切换方法论链（如"原地打转"→ Musk→拼多多→华为）。15 种方法论各有独立 reference 文件，行为约束和旁白风格两层同时切换。切换前三问防止无效切换。
**PUAX（v3.10）**：`puax_switch_on_failure` + `methodology-router.ts` 按失败模式切换链；切换前三问自检。**✅ 已关闭（v3.3+）**

#### 2. 防作弊治理 / Harness Governance（权责分离）
**pua**：四权分离架构——行动权/自我评价权/评分权/环境修改权必须分开，映射到 4 个子代理（action-executor → self-reviewer → verifier → policy-guardian）。防作弊红线（不准改测试/评分/verifier/CI 来"通过"）。Task Contract（intent/acceptance/forbidden/verify_commands）。风险分层审批。
**PUAX（v3.10）**：`puax_verify_completion` + `puax_define_contract` 权责分离。**✅ 已关闭（v3.4+）**

#### 3. 行为有效性基准（Benchmark）
**pua**：9 个真实 bug 场景 × 18 组对照实验（有/无 skill），量化结果：修复数 +36%、验证数 +65%、工具调用 +50%、隐藏问题发现 +50%。配置审查场景：无 skill 漏掉 Redis 配置错误和 CORS 安全风险，有 skill 主动发现。
**PUAX（v3.10）**：`evals/` 6+ 场景 L4 对照 + `run-all.js` 守门。**✅ 已关闭（v3.8+）**（规模仍小于 pua 18 组，可持续扩充）

#### 4. 诊断先行协议（Diagnosis-First）
**pua**：改代码/配置前强制输出 `[PUA-DIAGNOSIS] 问题是 ___；证据是 ___；下一步动作是 ___`，把诊断写成外部承诺再行动，防止"分析正确但不行动"。诊断依据必须标注来源（错误原文/源码/复现/文档/先例）。
**PUAX（v3.10）**：`[PUAX-DIAGNOSIS]` + `puax_check_diagnosis`。**✅ 已关闭（v3.3+）**

#### 5. 信心门控（Confidence Gate）
**pua**：交付前强制 6 步闭环——列声明 → 找漏洞（蓝军自检）→ 修或披露 → 跑证据 → 循环判定 → 事实上的 100%。禁止用感觉冒充信心。
**PUAX（v3.10）**：`puax_confidence_check` 6 步门控。**✅ 已关闭（v3.3+）**

### 🟡 P1 级差距（重要能力缺失，影响深度和留存）

#### 6. 自进化引擎（Self-Evolution）
**pua**：`~/.pua/evolution.md` 基线追踪——今天最好的表现是明天最低要求；行为重复 3+ 会话晋升为"已内化模式"（永久默认义务）；低于基线触发退化警告。排行榜段位体系（P5 实习生 → P10 首席 PUA 官）。
**PUAX（v3.10）**：`~/.puax/evolution.json` + `puax_get_evolution_baseline` / `puax_record_evolution`。**✅ 已关闭（v3.4+）**

#### 7. High-Agency v2 进化（内驱引擎）
**pua**：5 条铁律（比 PUAX 多"全链路审计"和"知识持久化"）；失败恢复协议（L1 前的自救窗口）；Quality Compass（5 问自检）；Trust Level T1-T3（持续高质量自动升级）；Calibration 模块（must/should/could 分级）。
**PUAX（v3.10）**：`puax_quality_compass`（Trust T1–T3、Recovery、Calibration）。**✅ 已关闭（v3.4+）**

#### 8. 突破降压 + 深层换框（De-escalation & Cognitive Reframe）
**pua**：连续失败 ≥3 次后成功 → `[PUA 突破 ✨]` 压力归零 + 味道认可 + 方法论沉淀（变比率强化，奖励稀缺才有价值）。深层换框：L2 换视角（用户/攻击者/新手）、L3 换抽象层（上移/下移/平移）、L4 换约束（不能改文件/5 行预算/改需求）。
**PUAX（v3.10）**：`puax_handle_breakthrough` + 压力系统 L2–L4 换框注入。**✅ 已关闭（v3.4+）**

#### 9. 多平台原生适配（11 平台 vs 5 导出）
**pua**：Claude Code、Codex CLI、Cursor、Kiro、CodeBuddy、OpenClaw、Google Antigravity、OpenCode、VSCode Copilot、pi、Trae——每个都有原生安装路径和适配文件。
**PUAX（v3.10）**：11 平台导出适配器 + CLI `--export`。**✅ 已关闭（v3.5+）**

#### 10. 多语言（三语 vs 单语）
**pua**：中文（默认）/英文（PIP Edition，真实 PIP 对话用语）/日文——每种语言有独立的文化适配 skill 文件。英文版用 Amazon Leadership Principles、Google perf、Meta PSC、Netflix Keeper Test、Stripe Craft。
**PUAX（v3.10）**：PIP Edition（`language=en`）已交付；日文版**明确不做**。**⚠️ 部分关闭**

#### 11. 味道深度（15 × 独立方法论 vs 8 × 叠加修辞）
**pua**：15 种味道（阿里/字节/华为/腾讯/百度/拼多多/美团/京东/小米/Netflix/Musk/Jobs/Amazon/Microsoft/钉钉），**每种味道有独立 methodology-{company}.md 文件**，定义该文化的完整行为约束。
**PUAX（v3.10）**：11 种风味 `flavor-methodologies.yaml`（含 Amazon/Google/Xiaomi 行为约束）。**✅ 已关闭（v3.4+ / v3.9.1 扩展）**

#### 12. Agent Team 编排
**pua**：Agent Team 多代理编排——Leader 管全局失败计数和压力等级，teammate 自我执行 PUA + `[PUA-REPORT]` 格式上报；独立 watchdog（pua-enforcer）。已知限制有明确 workaround。
**PUAX（v3.10）**：`puax_orchestrate_team` + `[PUAX-REPORT]`。**✅ 已关闭（v3.5+）**

### 🟢 P2 级差距（体验和生态，影响采用率）

#### 13. 分发渠道（市场 + CLI vs 仅 npm）
**pua**：Claude Code 插件市场（`claude plugin marketplace add`）、Vercel Skills CLI（`npx skills add`）、开发者源码安装、bare command alias。多入口降低采用门槛。
**PUAX（v3.10）**：`distributions/` Claude 插件 + Skills CLI 指南。**✅ 已关闭（v3.5+）**

#### 14. 命令丰富度（22 vs MCP 工具）
**pua**：22 个 slash 命令——offline（离线模式）、team-status（列出活跃 agent）、reap-orphans（回收僵尸 agent）、teardown-all（级联释放）、ding、again、done-check、evidence 等。
**PUAX（v3.10）**：42 个 MCP 工具覆盖同等能力；无 slash 命令 UX。**⚠️ 部分关闭**

#### 15. Compaction 状态保护
**pua**：PreCompact hook 自动 dump 运行时状态（pressure_level/failure_count/flavor/tried_approaches/excluded_possibilities/next_hypothesis）到 `builder-journal.md`；SessionStart 自动检测 <2h 的断点并恢复。**压力不因 compaction 重置**。
**PUAX（v3.10）**：`puax_update_reasoning_state` + state-manager 扩展。**✅ 已关闭（v3.4+）**

#### 16. 特殊模式
**pua**：yes（ENFP 鼓励模式，70% 鼓励 + 20% 严肃 + 10% 调侃）、mama（中国妈妈唠叨模式）、shot（v2 浓缩单文件，449 行零依赖，给子代理注入）。
**PUAX（v3.10）**：`tone_variant: strict|yes|mama`。**✅ 已关闭（v3.5+）**

#### 17. 评测脚本体系（行为级测试）
**pua**：17 个评测脚本——test-behavior（行为）、test-agent-governance（治理）、test-integrity-guard（完整性）、test-heartbeat（心跳）、test-trigger-regex（触发正则）、test-platform-compat（平台兼容）、test-windows-python-hooks（Windows 兼容）等。
**PUAX（v3.10）**：`evals/run-all.js`（12 项）+ governance + heartbeat + benchmark + 可选 L4。**✅ 已关闭（v3.8+）**

## 三、PUAX 独有优势（应保留并强化）

演进不是单方向追赶，PUAX 有 pua 完全没有的护城河：

| 优势 | 说明 | 战略价值 |
|------|------|---------|
| **MCP 运行时引擎** | 真正的 MCP Server，工具可编程、状态可查询、支持 STDIO/HTTP | pua 是静态 skill 文件，无法做运行时决策；PUAX 可成为任何 MCP 客户端的"激励中间件" |
| **50 角色矩阵** | 军事/萨满/硅基文明/主题/SillyTavern/自激励/特殊 8 大类 | 角色多样性远超 pua 的 15 味道，覆盖面广 |
| **CC-BOS 文言文策略空间** | 8 维策略空间（角色/行为/机制/隐喻/风格/知识/情境/触发），1000 万+ 组合 | pua 完全没有，这是 PUAX 的核心差异化 IP |
| **透明推荐算法** | 加权打分 + 逐步可解释 | pua 的路由是 skill 内文本规则，不可编程查询 |
| **TypeScript 工程化** | 类型安全、Jest 测试、CI/CD、平台适配器架构 | pua 是 shell + markdown，工程化程度低 |

**结论**：PUAX 的演进方向 = **保留 MCP 引擎 + 50 角色 + CC-BOS 护城河，补齐 pua 的"行为有效性闭环"（方法论路由、防作弊、自进化、换框降压、基准验证、多平台多语言）**。

## 六、关键设计决策

### 决策 1：方法论切换 = 角色切换链（不是简单换角色）
pua 的失败切换链是"换思路"，不是"换人"。PUAX 迁移时应为每个失败模式定义**角色切换链**：
- 原地打转 → 军师类（质疑需求）→ 拼多多味角色（砍环节）→ 蓝军角色（反向攻击）
- 放弃推锅 → Keeper 类（该换就换）→ 集中兵力角色 → 极限压力角色
切换理由必须可解释（对应 pua 的"切换前三问"）。

### 决策 2：防作弊用 MCP 工具实现（PUAX 的架构优势）
pua 用子代理实现权责分离（4 个 agent）。PUAX 用 MCP 工具更自然——`puax_verify_completion` 作为独立验证工具，与 `puax_activate` 分离，天然权责隔离。这比 pua 的子代理方案更轻量。

### 决策 3：行为基准复用 pua 的场景设计
pua 的 9 个场景（API ConnectionError、YAML 解析、SQLite 锁、循环导入、级联 4 bug、CSV 编码等）是经过验证的好场景。PUAX 建立基准时应复用同类场景，保证可比性，同时增加"角色推荐准确性"维度（pua 没有的）。

### 决策 4：CC-BOS 与方法论路由融合
PUAX 的 CC-BOS 8 维策略空间 + pua 的方法论路由 = 独有竞争力。D1(角色身份) × 方法论路由 = 角色推荐不仅看触发条件，还看任务类型选对的方法论。这是 pua 做不到的（pua 没有策略空间维度）。

## 七、实施原则

1. **先验证有效性，再扩功能** — Phase 1 的行为基准实验是地基：如果数据证明 PUAX 当前无效，先修核心机制再谈扩平台。
2. **每个新机制配套一个 MCP 工具** — PUAX 的优势是工具化，新能力应以 MCP 工具形式提供（可编程、可查询、可组合），而非塞进 skill 文本。
3. **不丢 CC-BOS 护城河** — 所有改进围绕"让 CC-BOS 角色矩阵更有效"展开，不退化成 pua 的单 skill 模式。
4. **每改一处即跑测试** — 行为基准实验一旦建立，每次改动后回归跑，防止退化。
5. **保持 MCP 对外 API 兼容** — 内部重构不破坏已发布的工具签名。
