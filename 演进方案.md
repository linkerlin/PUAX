# PUAX 项目审视与演进方案

> 审视日期: 2026-04-15 | 审视人: 造化宗师·Jobs  
> 当前版本: 3.2.0

## 总评

此项目理念有锋芒，50 角色、16 触发条件、Hook 系统——骨架完整。但骨架之下，**冗余堆积、重复滋生、架构分裂**。如同一台设计精妙但布满冗余零件的机器——能跑，但不美。

> **"简单是终极的复杂。"**

---

## 一、致命冗余：三份代码做同一件事

此乃最大之罪。

| 职责 | 文件 A | 文件 B | 文件 C |
|------|--------|--------|--------|
| 触发检测 | `core/trigger-detector.ts` | `core/trigger-detector-enhanced.ts` | `hooks/trigger-detector-enhanced.ts` |
| 反馈系统 | `core/feedback-system.ts` | `hooks/feedback-system.ts` | — |
| 工具处理 | `tools/*.ts`（定义+handler） | `handlers/*.ts`（另一套handler） | `mcp/handlers/*.ts`（第三套？） |
| 角色展示名 | `tools/get-role-with-methodology.ts` | `core/role-recommender.ts` | `handlers/trigger-handlers.ts` |
| `inferTaskType` | `tools/activate-with-context.ts` | `handlers/trigger-handlers.ts` | — |
| `TriggerDefinition` 类型 | `core/trigger-detector.ts` | `core/trigger-loader.ts` | `core/trigger-detector-enhanced.ts` |

**`handlers/` 和 `tools/` 做的是同一件事**——处理 MCP 工具调用。一个用 Zod schema + handler 对象，另一个用函数 + McpError。两条路并行，新人读代码如入迷宫。

**造化令**：合并 `tools/` 和 `handlers/` 为一套。删！删！删！

---

## 二、类型系统混乱

### `PuaxErrorCode` 命名不一致

```typescript
CONFIG_error      // 全大写_全小写
config_not_found  // 全小写_全小写  
CHAIN_error       // 全大写_全小写
tool_not_found    // 全小写_全小写
```

此非匠心之作。枚举值当统一为 `SCREAMING_SNAKE_CASE`。

### `TriggerDefinition` 定义三处

`core/trigger-detector.ts`、`core/trigger-loader.ts`、`core/trigger-detector-enhanced.ts` 各自定义了 `TriggerDefinition`、`TriggerPattern`、`TriggerDetection`。类型当统一定义于 `types.ts`，一处为真，处处引用。

---

## 三、假异步（Fake Async）

`core/role-mappings-loader.ts` 中：

```typescript
getTriggerRoleMapping(triggerId: string): Promise<TriggerRoleMapping | undefined> {
    return Promise.resolve(this.loadMappings().trigger_role_mappings[triggerId]);
}
```

8 个方法皆是同步操作包裹 `Promise.resolve`。此非异步，此乃**欺瞒**。要么真异步（文件 I/O 用 `fs/promises`），要么诚实同步。**不可两可。**

---

## 四、日志系统未贯彻

有 `utils/logger.ts`，但全项目仍有大量：

```typescript
console.log('[RoleMappingsLoader] Loaded role mappings');
console.warn('[Feedback] Initialization failed:', error);
console.error('[TriggerLoader] Error loading triggers:', error);
```

Logger 存在而不用，如利器在手而弃之。统一用 Logger，或删 Logger。

---

## 五、架构入口模糊

```
server.ts (8行) → server/core.ts (433行)
```

`server.ts` 仅是代理，真正逻辑在 `server/core.ts`。此 8 行文件何用？若为兼容，当注释说明；若无必要，当合并。

`index.ts` 调用链不清晰——启动时加载哪些模块、初始化顺序如何，全靠 `import` 副作用隐式执行。

---

## 六、`prompts-bundle.ts`：2549 行的巨物

此文件为生成产物，内含 50 个角色的完整 System Prompt。虽为生成，但其大小影响：
- IDE 加载缓慢
- Git diff 噪声巨大
- 每次改动一个角色，整个 bundle 重建

**建议**：按类别拆分为 `bundle-military.ts`、`bundle-shaman.ts` 等，按需加载。

---

## 七、`handlers/` 与 `mcp/handlers/` 的关系

`handlers/` 与 `mcp/handlers/` 内容相同。若为拷贝，当删其一。若为 symlink，当显式标注。**不可有二主。**

## 实施原则

1. **每改一处，即跑测试** — 不可批量改完再测
2. **先删冗余，再加功能** — 减法比加法更难，但减法才是真正的设计
3. **类型先行** — 统一类型定义后，再动逻辑
4. **保持兼容** — 对外 API 不变，只改内部结构
